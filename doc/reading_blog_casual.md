> 혼자 책 읽고 싶지만 토론도 하고 싶은 I형 개발자의 AI Agent 구축기

## 들어가며

저 MBTI I형이에요.

독서는 진짜 좋아하는데, 읽고 나면 항상 아쉬운 게 하나 있었어요. **누군가랑 이야기하고 싶다는 것.** 책이 던지는 질문들을 혼자 삭히기엔 너무 아깝잖아요. 그렇다고 북클럽이나 독서 모임을 찾아가자니 — I형인 저한테는 그 생각만 해도 이미 에너지가 반쯤 날아가요. 시간도, 장소도, 함께할 사람도 항상 아쉬웠죠.

그러다 어느 날 이런 생각이 들었어요.

> *"AI가 내 독서 파트너가 되면 어떨까? 언제든, 어디서든, 피곤하지 않게."*

마침 SW 엔지니어다 보니 그 생각을 바로 코드로 옮길 수 있었고요. 도구는 Anthropic의 Claude Code — 터미널에서 대화하듯 설계하고 구현하는 AI 코딩 도구예요. 이 글은 그 만들어가는 과정을 동료 개발자들이랑 나눠보려고 씁니다. **AI, 이렇게도 쓸 수 있거든요.**

## 설계 철학: Agent Harness

이 프로젝트에서 제일 중요한 개념이 **Agent Harness**예요.

하나의 거대한 AI 프롬프트로 다 처리하는 대신, **하나의 Main Agent가 여러 Sub Agent를 조율하는 구조**예요. Main Agent는 오케스트레이터 역할만 하고, 실제 일은 각자 전문화된 Sub Agent가 처리하는 거죠.

큰 그림은 이래요. **독서 에이전트가 그 첫 번째**고요.

```
Main Agent (오케스트레이터)
    │
    ├── 📚 독서 Agent          ← Agent #1 (현재 구현)
    │       │
    │       ├── Recommender    ← 이번 주 책 추천 (매주 월요일)
    │       ├── Insight        ← 책 → 삶의 지혜 연결
    │       ├── Memory         ← 독서 이력 분석
    │       ├── Trend          ← AI/기술 트렌드 연결
    │       │
    │       └── 💬 북클럽 봇  ← 핵심: 책 완독 후 실시간 1:1 토론
    │               ├── 완독 감지 ("고래 읽었어")
    │               ├── 다중 페르소나 토론 진행
    │               ├── 토론 요약 → 파일 저장 + git commit
    │               └── 7일 무완독 시 독촉 알림 발송
    │
    ├── ⛳ 골프 Agent          ← Agent #2 (예정)
    ├── 🤿 건강 Agent          ← Agent #3 (예정)
    └── 💰 재테크 Agent        ← Agent #4 (예정)
```

삶의 여러 영역에 에이전트를 하나씩 붙여가는 방식이에요. 에이전트가 늘어날수록 Main Agent가 종합하는 주간 리포트도 점점 풍부해지고요. 이 글에서 다루는 건 그 첫 번째 — 독서 에이전트예요.

각 Sub Agent는 **JSON으로만 소통**해요. 자연어를 배제하면 파싱 오류도 없어지고, 에이전트를 교체할 때 인터페이스 변경 없이 그냥 갈아끼울 수 있거든요.

```typescript
// types.ts — 에이전트 간 공유 계약
export interface InsightOutput {
  book: string;
  insights: Array<{
    domain: string;       // "삶의 지혜" | "경제/재테크" | "AI/기술" | ...
    application: string;  // 이 책의 아이디어가 이 영역에서 의미하는 것
    action: string;       // 이번 주 실제로 해볼 것
  }>;
}
```

이 구조가 좋은 이유가 뭐냐면, **에이전트 하나를 고쳐도 나머지에 전혀 영향이 없어요.** Insight Agent 프롬프트를 완전히 갈아엎어도 Main Agent는 그냥 똑같은 JSON만 받으면 되니까요.

## 시스템 구성

### 1. 주간 브리핑 (매주 월요일 08:00)

cron이 Main Agent를 실행해요. Main Agent는 Recommender를 먼저 돌리고, 결과를 바탕으로 나머지 세 에이전트를 **동시에** 실행하죠.

```typescript
// 순차 → 병렬로 바꾸는 것만으로 실행 시간 약 3배 단축
const [insights, memory, trends] = await Promise.all([
  runInsight(client, topRec.title, topRec.author, prefs),
  runMemory(client, history),
  runTrend(client, topRec.title, topRec.author, prefs.professional_interests),
]);
```

그러면 월요일 아침 8시에 텔레그램으로 이런 게 날아와요:

![주간 브리핑 텔레그램 메시지](images/briefing.jpg)

```
📚 이번 주 독서 브리핑 — 2026-05-19

📖 이번 주 추천
『사피엔스』 — 유발 하라리
→ 인지혁명과 AI 시대의 평행성 — 인간이 새로운 '허구'를 만드는 방식이 LLM 정렬 문제와 맞닿아 있음

💡 인사이트
[삶의 지혜] 집단적 믿음이 현실을 만든다는 통찰 → 팀 문화를 설계할 때 공유 내러티브의 힘을 의도적으로 활용해볼 것
[AI/기술]  인지혁명 = 언어로 허구를 공유하는 능력 → LLM이 '의미'를 다루는 방식과 동일한 메커니즘
```

### 2. 북클럽 AI 봇 (24시간 상시 대기)

이게 이 프로젝트에서 제일 핵심이에요.

책 다 읽으면 텔레그램에 그냥 보내면 돼요.

```
나:  "고래 읽었어"
```

봇이 자동으로 감지하고 토론을 시작해줘요.

```
봇:  🎙 고래 다 완독 축하드려요!

     고래는 압도적인 생명력과 폭력이 공존하는 소설이죠.
     춘희, 금복, 평화 — 세 여자의 이야기가 마치 신화처럼 펼쳐지는데,
     이 소설을 '여성의 이야기'라고 읽는 건 너무 편한 독해 아닐까요?
```

그 다음부터는 그냥 대화예요. 봇은 소크라테스식으로, 쉽게 동의 안 하고 계속 질문을 던지거든요.

![고래 토론 실제 대화](images/discussion.jpg)

```
나:  "바다에서 장사하던 시절 얘기가 인상적이었어.
     마치 장면이 상상이 되는 것 같아."

봇:  오, 흥미롭네요. 그 장면들이 '생생하게 상상된다'는 건
     천명관의 문체가 당신을 '목격자'로 자리잡게 만든 거예요.
     그 장면에서 묘하게 느껴지는 건 '자유'인가요, 아니면 다른 무엇인가요?

나:  "그런 것 같기도 하네. 나도 그런 자유를 원한 건가."

봇:  그렇죠, 근데 거기서 멈추면 안 돼요.
     그 자유가 구체적으로 어떤 자유인지 물어볼게요.

나:  "폭력적이고 착취적인 자유는 내가 원하는 게 아니야.
     그냥 삶의 구속감에서 벗어나고 싶은 건가 봐."

봇:  삶에 너무 힘이 들어가는 느낌, 쫓기는 기분 —
     그게 고래 속 인물들을 부러워하게 만든 거 아닐까요?
     슬로우라이프를 원한다는 건, 지금 삶에서 가장 속도를
     줄이고 싶은 게 뭔지 아세요?

나:  "토론 끝"

봇:  오늘 꽤 깊이 들어갔는데 —
     욕망의 진정성, 결핍의 정체성, 책임의 한계까지.
     근데 마지막 질문 피한 거 알고 있어. 😏
     나중에 다음 책 읽으면 또 와. 도망가도 질문은 기다리고 있을게.
```

#### 토론 파트너의 정체 — 다중 페르소나

봇이 단순히 "AI처럼" 대답하지 않는 이유가 있어요. 내부적으로 **4가지 전문가 페르소나**를 탑재하고, 대화 맥락에 따라 자연스럽게 전환하거든요.

| 페르소나 | 관점 | 특징 |
|----------|------|------|
| 🧠 심리학자 | 인간의 동기와 내면 | "그 선택의 무의식적 이유는 뭘까요?" |
| 🏛️ 철학자 | 존재·윤리·구조적 사유 | "그 전제 자체가 맞는 건가요?" |
| ⚙️ 엔지니어 | 시스템·인과·실용성 | "그게 실제로 작동하는 메커니즘은?" |
| 💼 사업가 | 가치·선택·실행 | "그래서 당신은 뭘 다르게 할 건가요?" |

같은 책이라도 어떤 페르소나가 질문하느냐에 따라 토론 방향이 완전히 달라져요. 철학자는 전제를 흔들고, 심리학자는 감정을 파고들고, 엔지니어는 구조를 묻고, 사업가는 행동을 촉구하죠. 혼자 읽었으면 절대 닿지 못했을 각도에서 책을 다시 보게 돼요.

토론이 끝나면 요약이 자동으로 **파일로 저장되고 git commit**돼요.

```bash
$ git log --oneline
abc1234 토론 요약: 고래 (2026-05-17)
def5678 토론 요약: 채식주의자 (2026-05-10)
```

`discussions/2026-05-17_고래.md` 파일이 생기고, 나중에 꺼내볼 수 있어요.

## 만들면서 삽질한 것들

AI 에이전트 처음 만들어보면 비슷한 실수를 하게 돼요. 저도 다 해봤고요.

### 실수 1. 잘못된 정보를 주면 AI는 틀린 방향으로 열심히 일한다

Insight Agent가 책 아이디어를 실생활에 연결하는 역할인데, `preferences.json`에 이렇게 해놨어요:

```json
{ "life_domains": ["골프", "프리다이빙", "강아지 키우기", "데일리 코디"] }
```

그랬더니 결과물이 이랬죠:

```
[골프]     카너먼의 시스템1/2 이론 → 티샷 전 3초 멈추기
[프리다이빙] 최악의 시나리오 심리 시뮬레이션
```

AI 입장에선 제가 준 정보대로 "열심히" 연결한 거예요. 근데 저한테 독서랑 골프는 완전 별개거든요. 취미 목록을 잘못 넣어놨으니 당연한 결과였죠.

그래서 이렇게 바꿨어요:

```json
{
  "reading_interests": ["삶의 지혜", "경제/재테크", "명상"],
  "professional_interests": ["AI/LLM", "시스템 설계"]
}
```

**배운 것**: "AI가 이상한 결과를 냈다"는 생각이 들면, 프롬프트보다 **내가 준 정보가 잘못된 건 아닌지** 먼저 의심해보세요.

---

### 실수 2. AI 특성을 무시하고 설계하면 나중에 구조를 통째로 뜯어야 한다

처음엔 Discussion Agent를 주간 브리핑 안에 넣었어요. 매주 월요일 아침에 "이 책의 토론 포인트"를 자동으로 생성해서 같이 보내는 방식이었죠.

근데 써보니까 바로 느껴졌어요. **토론이 일방통행이면 의미가 없잖아요.** 질문을 받아야 답이 나오고, 그 답을 들어야 다음 질문이 나오는 거니까요. 브리핑에 들어간 "토론 포인트"는 그냥 읽고 끝나는 텍스트였던 거예요.

결국 Discussion 전체를 뜯어서 독립적인 실시간 봇으로 다시 만들었어요.

```
Before: 브리핑 → 토론 포인트 포함 (일방적 출력)
After:  브리핑 → 추천만 / 봇 → 완독 감지 후 실시간 대화
```

**배운 것**: AI의 진짜 강점은 "생성"보다 "대화"에 있어요. 기능 설계할 때 AI와 사용자가 주고받는 구조인지, 일방적 출력인지를 먼저 정해두세요.

### 실수 3. 막히면 Claude Code한테 그냥 던져라

봇이 텔레그램 메시지를 받는 건 되는데, 응답이 ETIMEDOUT으로 계속 실패하는 거예요.

```
[bot] 수신: "고래 읽었어"
[bot] 오류: TypeError: fetch failed { code: 'ETIMEDOUT' }
```

curl로 테스트하면 되고, Node.js 직접 실행해도 돼요. pm2로 띄웠을 때만 안 되는 거죠. 재현 조건이 좁고 구글링으로도 답이 안 나오는 그런 류의 문제였어요.

그래서 Claude Code한테 현상을 그대로 설명했어요. 원인을 좁혀가더니, 각 가설 검증 방법을 제시하고, 해결책 코드까지 직접 짜줬어요.

원인은 WSL2의 IPv6/IPv4 DNS 처리 순서 이슈였고요:

```bash
# Node.js가 IPv4를 먼저 시도하도록
pm2 start dist/bot.js --node-args="--dns-result-order=ipv4first"
```
```typescript
// fetch → axios 교체
await axios.post(`${TELEGRAM_API}/bot${TOKEN}/sendMessage`, { ... });
```

저는 현상 설명하고 테스트 결과 보고하는 역할만 했어요. 디버깅이랑 코드 수정은 Claude Code가 다 했고요.

**배운 것**: AI 코딩 도구의 진가는 자동완성이 아니라 **막혔을 때** 나와요. 현상만 설명해도 같이 원인 좁혀가주거든요.

## 프로젝트 구조

```
my-ai-agent-v1/
├── src/
│   ├── agents/
│   │   ├── main-agent.ts          ← 주간 브리핑 오케스트레이터
│   │   ├── recommender-agent.ts   ← 책 추천
│   │   ├── insight-agent.ts       ← 삶의 지혜 연결
│   │   ├── memory-agent.ts        ← 독서 이력 분석
│   │   └── trend-agent.ts         ← AI/기술 트렌드 연결
│   ├── discussion/
│   │   ├── conversation-manager.ts ← 토론 세션 관리
│   │   └── summary-store.ts        ← 요약 저장 + git commit
│   ├── memory/
│   │   └── memory-manager.ts       ← JSON 기반 메모리
│   ├── bot.ts                      ← 텔레그램 봇 (24시간)
```

## 앞으로: Agent #2, #3, #4...

독서 에이전트 만들면서 확신이 생겼어요. **이 구조, 어떤 삶의 영역에도 붙일 수 있겠다**고요.

다음 에이전트는 이미 구상 중이에요.

| Agent | 하는 일 |
|-------|---------|
| ⛳ 골프 Agent | 라운드 기록 분석, 약점 패턴 감지, 주간 드릴 추천 |
| 🤿 건강 Agent | 프리다이빙 훈련 사이클 설계, 컨디션 기반 조언 |
| 💰 재테크 Agent | 시장 뉴스 요약, 포트폴리오 점검 알림 |

에이전트가 늘어날수록 Main Agent가 이들을 종합해서 **삶 전체를 조망하는 주간 리포트**가 되는 거예요. 각 영역을 AI가 조용히 tracking하다가 필요할 때 적절한 질문을 던져주는, 그런 개인 AI 시스템이요.

## 마치며

저한테 북클럽은 항상 '하고 싶지만 못 하는 것'이었어요. 에너지가 없어서, 시간이 안 맞아서, 장소가 멀어서.

이제는 텔레그램 열고 "고래 읽었어" 보내면 토론이 시작돼요. 자정에도, 새벽에도, 지하철에서도. 상대가 피곤하지도 않고, 스케줄 맞출 필요도 없고요.

AI를 검색이나 글쓰기 보조 정도로만 쓰고 계신다면, 이런 방식도 있다는 걸 보여드리고 싶었어요. **일상의 아쉬움을 에이전트 하나로 해소할 수 있다는 것.**

Claude Code는 그 과정에서 진짜 설계 파트너였어요. "이렇게 하면 어때?"를 코드로 바로 확인할 수 있는 그 속도감 덕분에, 생각이 실제로 완성까지 이어졌으니까요.

다음엔 골프 에이전트 만들어볼 생각이에요.
